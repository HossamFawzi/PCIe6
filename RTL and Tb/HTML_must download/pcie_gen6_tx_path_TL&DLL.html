USR_IF — User Logic Interface
واجهة المستخدم — باب الدخول الرسمي للـ TX Path
TX ENTRY
💡
الوظيفة الرئيسية / Main Function

يترجم طلبات الـ Driver إلى صيغة موحدة (604-bit packed word) ويمررها لـ REQ_Q. بيشتغل اتجاهين: TX (يبعت requests) و RX (يستقبل completions ويرجعها للـ Driver). كل الـ logic فيه combinational — مفيش flip-flops.

Translates Driver requests into a unified 604-bit packed word for REQ_Q. Pure combinational — no flip-flops. Also passes RX completions back to Driver.

MODULE · req_q.v
REQ_Q — Request Queue
طابور الطلبات — Dual FIFO منفصل للـ Posted و Non-Posted
FIFO
📦
ليه Dual FIFO وليس FIFO واحد؟ / Why Dual FIFO?

PCIe Spec يشترط إن Posted وNon-Posted يكونوا في queues منفصلة لأنهم بياكلوا credit classes مختلفة (PH/PD للـ Posted، NPH/NPD للـ Non-Posted). لو FIFO واحد: exhaustion في واحد يعطل التاني.

PCIe spec requires separate queues because they consume different credit classes. A single FIFO would cause head-of-line blocking.

هيكل الـ FIFO — FIFO Structure

// Parameters
parameter DEPTH_P  = 16;  // Posted FIFO depth
parameter DEPTH_NP = 16;  // Non-Posted FIFO depth
parameter WIDTH    = 576; // TLP descriptor width

// Routing Logic — يحدد P أو NP من req_type
wire [3:0] in_type   = req_in[575:572];  // req_type في top bits
wire       is_posted = (in_type == 4'd1); // MWr = Posted

// Full Detection
wire p_full  = ((p_wptr  - p_rptr)  == DEPTH_P);
wire np_full = ((np_wptr - np_rptr) == DEPTH_NP);

Read Priority — أولوية القراءة

⚖️
NP First Strategy

الـ REQ_Q بيفضل Non-Posted على Posted عند الـ dequeue (لو عندهم credit). السبب: NP requests محتاجة completion — تأخيرها بيزود الـ latency. ARB_TX بعدين يعمل round-robin بين الاتنين.

// Read side — priority: NP > P
if (!np_empty && credit_grant_np) begin
    req_out      <= np_mem[np_rptr];
    req_type_out <= 2'b01;  // Non-Posted
end else if (!p_empty && credit_grant_p) begin
    req_out      <= p_mem[p_rptr];
    req_type_out <= 2'b00;  // Posted
end

MODULE · tag_manager.v
TAG_MGR — Tag Manager
مدير الـ Tags — 1024 Tag Pool خاص بـ Gen6
⭐ GEN6
🏷️
PCIe Gen6 — 1024 Tags

Gen6 رفع الحد من 256 tag (Gen4/5) لـ 1024 tag. ده بيسمح بـ 1024 outstanding Non-Posted request في نفس الوقت — مهم جداً للـ high-bandwidth applications.

Gen6 increased tag space from 256 to 1024, enabling 1024 simultaneous outstanding NP requests.

Bitmap Allocator

reg [1023:0] free_bitmap;  // 1 = free, 0 = allocated

// Priority encoder — finds lowest free tag
function [9:0] find_free_tag;
    input [1023:0] bmap;
    integer i;
    begin
        find_free_tag = 10'd0;
        for (i = 1023; i >= 0; i = i - 1)
            if (bmap[i]) find_free_tag = i[9:0];
    end
endfunction

Tag Lifecycle — دورة حياة الـ Tag

ALLOCATION
  • يطلب TAG_MGR tag جديد عند كل NP request
  • يبحث في free_bitmap عن أول 1
  • يضع الـ bit = 0 (allocated)
  • يخرج tag_alloc + tag_valid
  • لو مفيش tags: tag_exhausted=1
RETURN / TIMEOUT
  • عند وصول Completion: tag_return_valid=1
  • يرجع الـ bit = 1 (free)
  • Timeout: لو الـ Completion أتأخر
  • Edge-detect منع الـ timeout من الـ fire كل cycle
  • outstanding_count يتحدث تدريجياً

FC Handshake — مصافحة التحكم في التدفق

قبل ما يتبعت أي TLP، لازم الجانبين يتفقوا على عدد الـ credits. ده بيتم عبر InitFC DLLPs. بعد اكتمال الـ handshake، fc_init_done=1 ويبدأ CR_MGR شغله.

CR_MGR — Credit Manager
مدير الـ Credits — يتحكم في كل TLP يخرج
FLOW CTRL
💳
6 Credit Counters

CR_MGR بيتابع 6 counters: PH, PD, NPH, NPD, CPLH, CPLD. مفيش TLP يعدي ARB_TX إلا لو عنده credit كافي. لو الـ receiver يرسل UpdateFC DLLP — يتم تحديث الـ credits.

Credit Consumption Rules

// Header credit: دايماً 1 لكل TLP
// Data credit:   = ceil(TLP_length / 4)  [1 credit = 4 bytes]

if (!tlp_is_np) begin  // Posted
    ph_avail <= ph_avail - 8'd1;
    pd_avail <= pd_avail - data_credits_needed;
end else begin         // Non-Posted
    nph_avail <= nph_avail - 8'd1;
    npd_avail <= npd_avail - data_credits_needed;
end

Infinite Credits — Credits اللانهائية

∞
لو الـ Receiver يعلن 0 credits = infinite credits

زي NP Data credits — الـ receiver دايماً عنده بالنسبة للـ NP data. CR_MGR بيرصد ده ومش بيحسبه في الـ grant condition.

Grant Logic

MODULE · arb_tx.v
ARB_TX — TX Arbiter
الـ Arbiter — يختار بين Posted وNon-Posted بـ Round-Robin
ARBITER
⚖️
Round-Robin Arbitration

لو الاتنين (P وNP) عندهم credit وجاهزين، ARB_TX يتبادل بينهم cycle وراء cycle عشان يمنع starvation. لو واحد بس جاهز — بيبعت المتاح. لو مفيش credit — يستنى.

Arbitration Logic

wire can_send_p  = req_p_valid  && credit_grant_p  && arb_ordering_ok;
wire can_send_np = req_np_valid && credit_grant_np && arb_ordering_ok;

if (can_send_p && !can_send_np)    // Posted فقط
    → select Posted
else if (!can_send_p && can_send_np) // NP فقط
    → select Non-Posted
else if (can_send_p && can_send_np) // الاتنين جاهزين
    → Round-Robin (last_granted ^= 1)

MODULE · ecrc.v

كاشف الأخطاء — CRC-32 على الـ Header والـ Data
CRC-32
🔒
ECRC — End-to-End CRC (PCIe 6.0 Section 2.7.1)

CRC-32 يُحسب على الـ Header والـ Data معاً. الـ EP bit يتفرض 0 أثناء الحساب. الـ ECRC يُلحق بالـ TLP كـ 32-bit DW، ويُعيّن الـ TD bit في الـ Header.

Polynomial: 0x04C11DB7 — نفس CRC-32 المستخدم في Ethernet.

TX Output — 1184 bits

// tlp_ecrc_tx[1183:0]
[1183:1152] = LTP DW        32b
[1151:1056] = LTP padding   96b
[1055:1024] = EETP DW       32b
[1023: 928] = EETP padding  96b
[ 927: 800] = TLP Header   128b  // TD bit = 1 إذا ecrc_en=1
[ 799: 288] = TLP Data     512b
[ 287: 256] = ECRC DW       32b  // القيمة المحسوبة
[ 255:   0] = padding      256b

CRC Algorithm

// Compute ECRC:
hdr_masked = hdr;
hdr_masked[112] = 1'b0;  // EP bit forced 0
crc = 32'hFFFF_FFFF;    // init
crc = crc32_over_header(hdr_masked, crc);
if (has_data)
    crc = crc32_over_data(data, crc);
ecrc = reflect32(crc) ^ 32'hFFFF_FFFF;  // final XOR

Flit Mode — مميزة حصرية لـ PCIe Gen6

Gen5 وما قبلها: كل TLP يُرسل منفرداً مع 128b/130b encoding. Gen6: TLPs تُحزّم في Flits بحجم ثابت 256 byte (2048 bit) مع CRC-24 خاص بكل Flit. ده بيقلل الـ overhead ويزود الـ efficiency لـ ~98%.
